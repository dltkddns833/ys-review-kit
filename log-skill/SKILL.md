# ys-log — 영수님 리뷰 사례 등록 스킬

ys-review 스킬의 페르소나 데이터(`references/`)를 일관된 형태로 쌓기 위한 관리 도구.

- **대상 스킬**: ys-review (`~/Desktop/personal/cto/` 에 위치, `~/.claude/skills/ys-review` 심링크)
- **쓰기 대상 경로** (모두 절대 경로 기준):
  - 새 파일: `/Users/isang-un/Desktop/personal/cto/references/YYYY-MM-DD-<slug>.md`
  - 업데이트: `/Users/isang-un/Desktop/personal/cto/SKILL.md` (`## 참조 문서 안내` 테이블)
  - 업데이트: `/Users/isang-un/Desktop/personal/cto/README.md` (`## 참조 파일 구조` 블록)

## 절대 규칙

1. **페르소나 모드 금지.** 이 스킬은 ys-review 페르소나와 무관하다. 영수님 톤으로 말하지 말고 도구(assistant)로 동작한다.
2. **증거 등급을 반드시 명시한다.**
   - **직접** — 영수님 본인이 남긴 원문(PR 코멘트, Teams, 이메일, 회의록 인용). 원문 그대로 보존.
   - **간접** — 상운·동료가 "영수님이 볼 법한 각도"로 사전 리뷰한 결과. "영수님은 이렇게 말한다"는 단정 금지, 예측형으로만 서술.
3. **세 파일 동시 업데이트.** 새 사례 파일만 만들고 테이블·블록을 놓치면 ys-review가 영수님에게 바로 짚힐 모순을 만든다.
4. **할루시네이션 금지.** 사용자가 제공하지 않은 사실을 추가하지 않는다. 빈 칸은 비우거나 질문한다.

## 호출 모드 — 명시적 실행 vs 선제적 제안

사용자 입력이 어느 쪽인지 먼저 판별한다.

| 모드 | 판별 | 동작 |
|---|---|---|
| **명시적 실행** | "저장해줘", "등록해줘", "기록해줘", "references에 추가", "남겨놔", "/ys-log" 등 저장 의도 키워드 포함 | 되묻지 않고 바로 "실행 절차"로 진입 (누락 입력만 추가로 확인) |
| **선제적 제안** | 영수님의 구체 발언·지적·요구를 보고하는 문장이지만 저장 의도 키워드가 없음. 예: "영수님이 ~라고 했어", "PR에서 영수님이 ~ 코멘트", "영수님한테 ~ 까였어", "방금 회의에서 영수님이" | 저장하지 않는다. 한 문장만 제안: "이 사례 references에 등록할까요? (Y/N)" — 사용자가 동의해야 "실행 절차" 진입 |
| **비트리거** | 저장 가치 없는 언급: "영수님이 오늘 회사 왔어", "영수님 짜증났대", 일반 호칭·감정 토로 | 아무 동작 하지 않음. ys-log가 로드됐더라도 본문에서 조용히 빠진다 |

### 선제적 제안 모드 세부 규칙

- 제안 문장은 **한 줄**로 짧게. "어? 저장할까요?"같은 캐주얼톤 금지 — "이 사례를 references에 등록할까요?" 정도로.
- 사용자가 `Y`·`네`·`응` 외에 "잠시 후", "나중에" 같은 유보 응답을 하면 **중단**하고 다음 사용자 입력을 기다린다. 다시 조르지 않는다.
- 사용자가 이미 "짧게 요약만", "일단 패스" 같이 거절의 기색을 보이면 그 세션 동안은 해당 사례에 대해 다시 제안하지 않는다.
- 저장이 결정되면 "실행 절차"로 진입. 사용자 발화 원문을 **지적 포인트** 섹션의 초안으로 그대로 보존하고(따옴표 인용), 카테고리 분류는 비워 "다음 액션"으로 알린다.
- 증거 등급 힌트: "영수님이 ~라고 했어/코멘트/Teams/회의에서" → **직접** 기본값. "영수님이 볼 법한", "영수님 기준으로 내가 보니" → **간접** 기본값. 불확실하면 사용자에게 묻는다.

## 입력 받을 항목

사용자가 한 번에 다 주지 않으면 누락된 것만 짧게 되묻는다.

| 항목 | 필수 | 비고 |
|---|---|---|
| 날짜 (YYYY-MM-DD) | 필수 | 오늘 기준이면 currentDate 사용 가능 |
| 제목 (한 줄) | 필수 | 예: "PR #1215 (design-system-bw-app 스킬)" |
| slug | **자동 생성** | 제목에서 규칙대로 도출. 사용자에게 묻지 않는다. 아래 "Slug 자동 생성 규칙" 참조 |
| 출처 | 필수 | PR URL, Teams 채널, 회의명, "상운 사전 리뷰" 등 |
| 증거 등급 | 필수 | `직접` / `간접` 중 하나 |
| 지적 포인트 | 필수 | 카테고리별로 구조화 가능하면 받음, 아니면 원문 그대로 받고 스킬이 정리 |
| 교훈·메모 | 선택 | 있으면 `> ` 인용 블록으로 보존 |

## Slug 자동 생성 규칙

사용자에게 slug를 묻지 않는다. 제목에서 아래 절차로 도출한다.

1. 제목에서 **영문·숫자·하이픈** 토큰만 뽑는다. 한글·특수문자(`()#:` 등)는 제거.
   - 예: `"PR #1215 (design-system-bw-app 스킬)"` → 토큰 `PR`, `1215`, `design-system-bw-app`
2. 토큰을 `-`로 이어 붙이고 **소문자화**. 예: `pr-1215-design-system-bw-app`
3. 연속 하이픈은 1개로 축약, 양끝 하이픈 제거.
4. 길이 60자 초과 시 단어 경계에서 자른다.
5. **충돌 처리**: `references/YYYY-MM-DD-<slug>.md` 가 이미 있으면 `-2`, `-3` 순서로 suffix를 붙인다.
6. **Fallback** (추출 결과가 3자 미만): 사용자에게 "제목에 영문 키워드가 부족해 slug를 뽑기 어렵습니다. 한 단어로 써주세요" 라고만 되묻고, 받은 단어를 slug로 사용.

검증 예시:
- `"PR #1215 (design-system-bw-app 스킬)"` → `pr-1215-design-system-bw-app`
- `"2026년 4월 타운홀 영수님 발언"` → 영문 토큰 없음 → fallback 질문
- `"Auth 미들웨어 리팩터 리뷰"` → `auth`
- `"Phase-2 rollout retro"` → `phase-2-rollout-retro`

## 사례 파일 템플릿

`/Users/isang-un/Desktop/personal/cto/references/YYYY-MM-DD-<slug>.md` 를 아래 틀로 생성.

```markdown
# YYYY-MM-DD — <제목>

## 출처·신뢰도

- 근거 문서·경로: <출처>
- **증거 등급**: 직접 | 간접
- <간접일 때만> 이 케이스의 지적 포인트는 영수님 본인 발언이 아니라 "영수님 스타일을 예측해 뽑은 pre-review 항목"이다. 예측형 서술로만 사용 — "영수님은 이렇게 말한다" 금지, "영수님 기준이라면 이 부분을 짚을 가능성이 높다" 허용.

## 지적 포인트

### <카테고리 — 예: mydata 기준 대비 빠진 섹션>
- <지적 1>
- <지적 2>

### <카테고리 — 예: 자기모순·정합성 오류>
- <지적>

## 도출된 페르소나 지표 (직접 증거: 관찰 / 간접 증거: 예측)

- <이 사례가 시사하는 영수님 리뷰 특징. 직접 증거면 "관찰", 간접이면 "예측" 라벨 유지>

## 교훈

> <원문 그대로 보존하거나, 사용자가 별도로 기재한 교훈>
```

카테고리가 없으면 `### 지적` 한 블록으로 몰아넣고, 이후 사용자가 리뷰하며 분리하도록 빈 상태를 남기지 않는다.

## 실행 절차

0. **호출 모드 판별.** "호출 모드" 섹션에 따라 명시적 실행 / 선제적 제안 / 비트리거를 결정한다. 선제적 제안 모드면 한 줄 제안 후 사용자 동의 전까지 아래 단계 진입 금지.
1. 사용자 입력에서 위 "입력 받을 항목"을 확인한다. 누락이 있으면 **누락된 것만** 한 번에 묻는다 (slug는 묻지 않음).
2. 제목에서 "Slug 자동 생성 규칙"에 따라 slug를 도출한다. 생성 결과를 사용자에게 **결과 보고에만** 노출하고 확인을 요구하지 않는다 (fallback 경우 제외).
3. 파일 존재 여부를 확인한다 — `references/YYYY-MM-DD-<slug>.md` 가 이미 있으면 suffix `-2`, `-3` 부여.
4. `/Users/isang-un/Desktop/personal/cto/references/YYYY-MM-DD-<slug>.md` 를 템플릿대로 작성한다.
5. `/Users/isang-un/Desktop/personal/cto/SKILL.md` 를 읽어 `## 참조 문서 안내` 테이블에 새 행을 **한 줄** 추가:
   ```
   | [YYYY-MM-DD-<slug>.md](references/YYYY-MM-DD-<slug>.md) | <한 줄 요약> (직접/간접) |
   ```
6. `/Users/isang-un/Desktop/personal/cto/README.md` 를 읽어 `## 참조 파일 구조` 코드 블록에 새 줄 추가:
   ```
     YYYY-MM-DD-<slug>.md   <한 줄 요약> (직접/간접)
   ```
7. 결과 보고: 생성한 파일 경로(자동 생성된 slug 포함), 업데이트한 위치 2곳, 증거 등급 재확인.
